## Принципы классификации

### 1. Лист = единица фильтрации
Каждое требование попадает на один лист дерева `Домен → Подкатегория → (при нужде) Суб-суб-категория`. Карточка вакансии/кандидата собирается как *множество активных листьев*.

### 2. Платформа-экосистема = сквозной тег, а не домен
Apple / Google (Android) / Microsoft / AWS / Azure / IBM / Oracle / SAP / Atlassian / Apache / Яндекс — это **теги**, накладывающиеся поверх листьев. Конкретные инструменты платформы живут в своих функциональных доменах:
- AWS Lambda → DevOps/Cloud + тег `AWS`
- UIKit → Development/Mobile/iOS + тег `Apple`
- Яндекс.Метрика → Analysis/Продуктовая аналитика + тег `Яндекс`
- Firebase → Development/Mobile/SDKs + тег `Google`

Это позволяет на карточке показывать и функциональный профиль, и экосистемный срез.

### 3. Исключения: автономные экосистемы
Две экосистемы слишком велики и самодостаточны, чтобы размазывать по доменам. Они вынесены отдельным субдоменом в **Разное**:
- **1С** — свой язык, своя аналитика, своя разработка, своё тестирование, свои конфигурации
- **Bitrix / Bitrix24** — CMS + CRM с собственной архитектурой разработки и внедрения

### 4. Разделение внутри домена
В каждом домене встречается три типа требований:
- **Концепции** (дольше живут, определяют школу мышления: принципы, методологии, нотации, архитектуры, паттерны)
- **Инструменты** (конкретные продукты, библиотеки, SDK — быстрее устаревают)
- **Опыт** (отраслевой, масштаб, тип задач, уровень)

Раздел `## Инструменты` в каждом файле — главный практический объект для фильтрации карточек.

### 5. Пересечения доменов
Некоторые навыки живут на стыке двух доменов. Они размещены в «основном» домене, но явно упомянуты в разделе `## Пересечения` обоих доменов. Примеры:
- SQL как язык → Development/БД, но применяется в Analysis и QA
- CI-пайплайны автотестов → QA/Авто + DevOps/CI
- Безопасная разработка → Development + InfoSec
- Вёрстка → Design + Development/Frontend
- Архитектура решений → Development + Analysis

### 6. Гранулярность инструментов
Инструменты группируются по назначению (например, «Брокеры сообщений», «ETL-инструменты», «BI»), а не по поставщику. Это даёт фильтр вида «любой инструмент для X».

## Автономные субдомены в «Разное»

- `1С` — полная экосистема: платформа, язык, конфигурации, инструменты, аналитика, тестирование
- `Bitrix` — платформа + Bitrix24 (CMS, интернет-магазин, CRM)
- `Gamedev / графика` — движки, графические API, шейдерные языки (пересекает Development)
- `Embedded / IoT` — микроконтроллеры, промышленные протоколы (пересекает Development)
- `Blockchain / Web3` — свой стек смарт-контрактов и кошельков
- `Telecom-стек` — 3GPP/4G/5G/SS7/Diameter и т.п.

## Как собирается карточка

Карточка вакансии или кандидата — это *срез* реестра:

```
[Тег платформы: Apple, Google]
Development:
  - Языки: Kotlin, Swift
  - Mobile/Android: Jetpack Compose, Coroutines, Room
  - Mobile/iOS: SwiftUI, Combine, CoreData
  - Архитектура: MVVM, Clean Architecture
  - БД/NoSQL: Realm, SQLite
QA:
  - Unit/UI тесты
DevOps:
  - CI: Fastlane, GitLab CI
Разное:
  - Soft: Agile/Scrum, Code review
  - Отрасль: FinTech
```

Фильтр работает в две стороны:
- **Вакансия → кандидат**: ищутся кандидаты, чей срез покрывает M обязательных листьев вакансии
- **Кандидат → вакансия**: показываются вакансии, чьи обязательные требования пересекаются со срезом кандидата более чем на N %

## Статус реестра

Это первый полный проход. В каждом файле:
- **Полные подкатегории** — структура зафиксирована
- **Инструменты** — собраны из первичного бульона (не исчерпывающе, но репрезентативно)
- **Метки неоднозначности** `⚠️` — помечают листья, классификацию которых стоит пересмотреть
- **Пересечения** — явно перечислены в конце файлов
